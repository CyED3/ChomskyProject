# Literature Review

Before implementing Chomsky, a literature review was conducted to understand the intersection between Formal Language Theory and Software Security, specifically in the context of static analysis and secret detection.

## 1. Secret Scanning in Industry

The problem of hardcoded secrets (API keys, passwords, database URIs) is well documented. Tools like **TruffleHog**, **git-secrets**, and **Gitleaks** represent the industry standard for this task. 

These tools operate almost exclusively at the **lexical level** using Regular Expressions (Type-3 grammars in the Chomsky Hierarchy). While highly efficient, regex matching suffers from two primary limitations:
- **High false positive rates**: Regular expressions lack context. They struggle to differentiate between a safe environment variable assignment and a hardcoded token.
- **Lack of statefulness**: Regex cannot track temporal relationships (e.g., matching a credential definition on line 10 and a print statement exposing it on line 50).

Chomsky addresses these limitations by introducing Finite Automata (DFA) on top of the regex layer, allowing the system to track state and confirm leaks contextually.

## 2. Formal Language Theory in Static Analysis

Static Application Security Testing (SAST) tools rely heavily on Abstract Syntax Trees (ASTs) generated by parsers. Context-Free Grammars (CFGs, Type-2 grammars) are fundamental to this process.

As noted in standard formal languages literature (e.g., *Introduction to the Theory of Computation* by Michael Sipser), nested structures with matched brackets (like `{}` in configuration files) cannot be processed by finite automata. The Pumping Lemma for Regular Languages proves this limitation. Therefore, Chomsky incorporates a CFG validator (Module 4) to ensure the syntactic correctness of configuration files, enforcing that they conform to security policies at the structural level.

## 3. Finite State Transducers (FSTs) for Automated Refactoring

While much research focuses on *detecting* vulnerabilities, automated remdiation is an ongoing challenge. Finite State Transducers (FSTs) extend deterministic finite automata by mapping input strings to output strings. 

In Chomsky, FSTs are used to formalize the refactoring process (Module 3). By modeling code transformation as a transduction problem, the tool can systematically propose secure alternatives to insecure constructs (e.g., transforming a plaintext assignment into an OS environment variable reference). The non-determinism inherently present in physical implementations of some FST paths (e.g., deciding whether to flag or passthrough an IP) allows the system to establish security priority queues.

## Conclusion

The architecture of Chomsky maps directly to the Chomsky Hierarchy of formal languages:
1. **Regular Expressions (Lexical)**: Detects basic structural violations.
2. **DFA (Behavioral)**: Tracks sequences of events to confirm breaches.
3. **FST (Transduction)**: Proposes automated safe transformations.
4. **CFG (Syntactic)**: Validates recursive, nested configurations safely.

This unified theoretical approach guarantees deterministic, mathematically verifiable behavior when auditing repository files for security vulnerabilities.
